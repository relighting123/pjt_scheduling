# TOOL_CHANGE_BENCH — tool 교체(전환) 대표 스케줄링 문제 10종 + 정답지

반도체 라인에서 실제로 마주치는 "재공 상황별 tool 교체" 대표 시나리오 10개를
`data/dataset/`에 생성하고, 각각에 대해 **구성적으로 증명한 최적해(정답지)**를
`docs/tool_change_bench_meta.json`에 기록한다. `earliest_st` / `min-progress`
두 휴리스틱이 그 정답지에 실제로 얼마나 도달하는지 `docs/tool_change_bench_results.json`
에 담아 비교한다.

## 실행 방법

```bash
python benchmark/gen_tool_change_bench.py      # 10개 데이터셋 생성 → data/dataset/TCB*
python benchmark/tool_change_bench.py --verify # 알고리즘 비교 + 오라클 정책으로 정답지 재검증
```

`--verify`를 붙이면 각 데이터셋마다 손으로 설계한 "오라클 정책"(스크립트로 짠 정확한 스케줄)을
시뮬레이터에 직접 돌려서, 정답지에 적어둔 생산량·전환 횟수가 **실제로 달성 가능한지**
재확인한다. 7개 데이터셋(TCB04~10) 모두 `✓ 일치`로 확인됨 — 아래 결과는 종이 위 계산이
아니라 프로젝트의 실제 `SchedulingSimulator`로 검증된 값이다.

## 3가지 대표 유형

| 유형 | 문제 상황 | 데이터셋 |
|------|-----------|----------|
| **A. 단일 제품·공정별 재공 충분** | 제품 1개, 여러 공정이 각각 전담 장비 + 충분한 초기 재공 보유. 전환이 아예 필요 없는 대조군 | TCB01~03 |
| **B. 다품종·공정별 재공 충분, 장비 공유** | 제품 여러 개가 2단계(OPER001→OPER002) 공정에서 장비를 공유. 제품 수>장비 수, 물량 편중, 전환시간≫처리시간 등 | TCB04~07 |
| **C. 재공 편중 + 안전재공 확보 후 전환** | 한 제품이 OPER001(재공 다량)→OPER002(재공 0, OPER001 완료로만 유입)로 흐르고, 두 공정 겸용 장비가 있음. 단순 진행률 균형 로직은 트리클 유입마다 전환해버리는 "전환 쏠림"이 발생 | TCB08~10 |

카테고리 C가 사용자가 설명한 "단순히 takt time으로 하면 잦은 전환이 발생하여 안전재공을
확보할 때까지 특정 공정을 진행하다가, 후공정에 재공이 충분히 모이면 전환하는" 문제를
정확히 재현한다.

| TCB09_SAFETY_2EQP_BUFFER | C | 겸용 장비 2대, 재공 60→계획 40 | 100 / 1 | 96/100, 2전환 | 89/100, 3전환 |

**집계(정답지 대비)**: Earliest-ST 생산률 97.2%(10개 중 6개 정답지 도달), Min-Progress
생산률 89.9%(10개 중 5개 도달). 두 알고리즘 모두 카테고리 C(안전재공)에서 정답지 대비
격차가 가장 크다 — 특히 TCB10에서 Min-Progress는 정답지 1회 대비 **14회 전환**하며
production 120 중 70만 달성(58%)해, "잦은 전환" 실패 패턴이 뚜렷이 드러난다.

## 최적해를 어떻게 구했나 (정답지 도출 방법)

단순히 수식으로 우겨넣은 값이 아니라, **실제로 달성 가능한 구성적(constructive) 스케줄**을
손으로 설계해 도출했다. 카테고리별 방법:

- **A(단일 제품·전담)**: 장비를 공정별로 1:1 전담시키면 전환이 원천적으로 불필요.
  장비수×⌊horizon/ST⌋가 곧 상한이자 달성 가능한 값 — 자명한 증명.
- **B(다품종·장비 공유)**: "장비 수 vs 제품 수" 비둘기집 하한(`max(P-E, 0)`회 전환)과
  장비별 담당 제품을 블록으로 묶어 makespan을 손계산 → 그 makespan을 `sim_end_minutes`로
  설정해 하한과 달성 가능한 구성이 동시에 성립하도록 만들었다. TCB05/07은 장비별 담당
  제품군을 **전용 장비모델**로 고정해(예: EQP1↔{A,B}) 의도치 않은 교차 배정을 원천
  차단했다.
- **C(재공 편중·안전재공)**: 겸용 장비 k대 중 일부를 상류(up) 공정에서 안전재공이 쌓일
  때까지 함께 진행시키다가, 균형점(crossing-point) 시점에 하류(down) 담당 장비만 전환해
  영구적으로 유지하는 정책의 makespan을 직접 계산했다(1EQP는 완전 증명, 2·3EQP는
  균형점 구성으로 이론 하한 대비 2% 이내로 근접).

### 검증: 오라클 정책으로 실제 시뮬레이터 재현

각 정답지에 대해 그 스케줄을 그대로 수행하는 "오라클 에이전트"(`benchmark/tool_change_bench.py`의
`SafetySwitchOracle`, `BlockDedicationOracle`)를 만들어 실제
`SchedulingSimulator`를 구동했다. 개발 중 이 과정에서 실제로 두 가지 버그를 발견하고
고쳤다 — 종이 계산만으로는 드러나지 않았을 문제들이다.

1. **idle EQP 지연 해석**: `env.step()`이 idle 장비를 지연 해석(lazy resolve)하기 때문에,
   오라클이 액션을 계산하는 시점에 `current_idle_eqp()`가 아직 `None`이면 이전 라운드의
   낡은 액션이 엉뚱한 장비에 적용될 수 있었다. `env._ensure_decision_eqp()`를 먼저 호출해
   해석을 강제하는 것으로 해결.
2. **유휴 장비의 의도치 않은 개입**: 프레임워크는 "가능한 행동이 있으면 반드시 뭔가
   배정"하는 방식이라(`_resolve_ppk_oper`가 항상 `feasible[0]`로 폴백), 자기 몫을 일찍
   끝낸 장비가 남의 제품을 부분적으로 도와버려 정답지보다 많은 전환이 발생했다. 실제
   손실은 없었지만(총생산은 상한과 일치) 전환 수가 부풀려졌다 — `max_conversions_per_eqp`로
   장비당 전환 상한을 걸어(TCB04/06) 혹은 제품군을 전용 장비모델로 고정해(TCB05/07)
   봉쇄하고, 0전환·최소전환 정답지가 실제로 달성됨을 확인했다.

## 알고리즘 vs 정답지 격차가 시사하는 것

- **카테고리 A**: 두 알고리즘 모두 100% 달성 — 전환 유인이 없는 문제에서는 차이가 없다.
- **카테고리 B**: Min-Progress가 대칭(TCB04)에는 강하지만 제품 과잉·전환과중(TCB05/07)에는
  약함(진행률만 보고 전환 비용을 고려하지 않아 잦은 왕복 발생). Earliest-ST는 반대 경향.
- **카테고리 C**: 두 알고리즘 모두 정답지에 못 미치며, 특히 Min-Progress가 심각하게
  저조(TCB10: 58% 생산, 정답지 대비 14배 전환). 두 알고리즘 모두 "안전재공이 쌓일 때까지
  기다렸다가 한 번만 전환"하는 정책을 스스로 찾아내지 못한다. `config.py`에는 이미
  `flow_balance_starving_cover_min`(후속 공정 커버 시간 기준 starving 판정)과
  `w_flow_balance` 보상 항이 마련돼 있지만, 현재 기본값은 `w_flow_balance=0.0`으로
  비활성 상태다(주석: "전담 방해 2위라 제거"). 이 벤치마크가 보여주는 격차는 그 트레이드오프
  — 안전재공을 고려한 보상을 켜면 전담이 흔들릴 위험 vs. 끄면 카테고리 C처럼 전환이
  폭주할 위험 — 를 수치로 드러낸다. RL 에이전트로 이 10종을 학습·평가하면 이 트레이드오프를
  구체적으로 튜닝하는 기준선으로 쓸 수 있다.

benchmark/gen_tool_change_bench.py   # 10개 데이터셋 생성기 (data/dataset/TCB01~10)

커밋되지 않는다 — `python benchmark/gen_tool_change_bench.py`로 언제든 재생성한다.
